# PM Feedback 文件骨架（前端視角）

寫入路徑：`requirement-feedback/<slug>-frontend.md`

> 這份文件給 PM 看。**全文使用業務語言，不出現 RD 術語**。
> 禁用詞：component / state / route / props / hook / SSR / hydration 等等。

---

```markdown
# 需求 Audit Feedback — Frontend 視角

- 需求名稱：<從 PM 文件標題或 slug 推>
- Audit 日期：<YYYY-MM-DD>
- 來源：<PM 文件路徑或「貼文字」>

## 一句話摘要

<把 PM 給的需求重述為一句話，讓 PM 確認 RD 的理解沒走偏>

## 整體觀察

<用 2-3 句話陳述：哪幾個維度給得清楚、哪幾個維度需要補。**不寫機器可解析欄位**，純自然語言。>

範例：
> 這份需求的「使用者要做什麼」與「主要畫面」寫得清楚，但「畫面細節」與「操作後的回饋」這兩塊還需要補一些細節。整體看 6 個重點裡有 3 個給得足，3 個需要 PM 再補一輪。

## ✅ 已給足的部分

逐項列點。每點一句陳述。

例如：
- 主要使用者角色（一般會員、付費會員）已點名清楚
- 進入這個功能的入口（首頁推薦、會員中心）已列出
- 主要畫面（列表頁、詳情頁、確認頁）已盤點

## 🟡 需要補完的部分（精簡版）

每個維度一段，列出該維度為什麼判定為「需補」，**不重複下方逐題清單**。

例如：
- **互動細節**：每個畫面上有哪些按鈕 / 輸入框，以及點下去後預期發生什麼，目前只描述了主要按鈕，次要的（排序、篩選、tab 切換）沒提到。
- **畫面狀態**：「資料載入中」「沒有資料時」「載入失敗時」這三種狀態畫面長怎樣未說明。
- **重新整理行為**：使用者按 F5 後，原本選好的篩選條件還在嗎？沒有交代。

## 💡 進階建議

未來可以這樣準備需求單，會讓後續開發更順：

例如：
- 寫需求單時可以附一張「畫面之間怎麼跳」的流程圖
- 對每個畫面準備一張「五種狀態」表（載入中、有資料、空、失敗、成功）
- 提到網址時，順便寫「網址要不要包含篩選條件以便分享」

## 補完後的下一步

1. PM 在每個「PM 補答」區塊填上回答
2. 把補完的版本貼回對話，或更新原 PM 需求文件
3. 重跑 audit：`/athena-dev-plugin:athena-audit-requirement-frontend <path> --slug=<slug>`
4. 若整體變綠（多數維度 ✅），即可進入前端 RD 工程流程（`/athena-dev-plugin:athena-point` 評分後決定走 spec 或 build）

---

> 這份回饋是輔導建議，不阻擋任何工程流程。即使有題目沒補完，前端 RD 仍可以憑現有資訊開工，只是會在 build 過程中多回頭問 PM 幾次。
```
