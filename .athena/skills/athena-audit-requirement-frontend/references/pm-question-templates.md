# PM-Friendly 提問模板（前端視角）

這份模板是給 PM 看的澄清問題範本。**全部用業務語言**，不出現 RD 術語。
寫 PM feedback doc 時，依當下需求改寫具體名詞（把「使用者」「商品」等替換成 PM 文件實際的對象）。

## 寫作原則

| 原則 | 範例 |
|------|------|
| **用業務語言** | ❌「component 沒拆」→ ✅「畫面上的『商品卡片』要不要做成可重複用的版型？」 |
| **問「行為」而非「實作」** | ❌「state 怎麼管」→ ✅「使用者切換 tab 後再切回來，原本的篩選條件還在嗎？」 |
| **附上「為什麼問這題」** | 讓 PM 理解這題不是吹毛求疵，是真的會卡到 |
| **留空白讓 PM 直接補** | 在問題下方留「PM 補答：____」 |

## 禁用詞

`component` / `state` / `route` / `routing` / `SSR` / `SSG` / `hydration` / `props` / `hook` /
`store` / `dispatch` / `event listener` / `re-render` / `mount` / `unmount` / `lazy load` /
`code split` / `bundle` / `i18n key`

替代寫法：

| RD 術語 | PM-friendly 說法 |
|---------|------------------|
| component | 「畫面元素」「重複出現的版型」 |
| state | 「畫面狀態」「目前選了什麼」 |
| route / routing | 「網址」「頁面」「畫面切換」 |
| props | 「傳給這個版型的資料」 |
| event listener | 「點下去之後做什麼」 |
| state persistence | 「重新整理後還記得嗎」 |
| URL params | 「網址裡帶的資訊」「能不能複製連結分享」 |

## 維度 1：Actor / Persona Identification（角色 / 使用者識別）

### Q1.1 — 誰會用這個功能
> 哪些角色會使用這個功能？（例如：一般使用者、付費會員、管理員、客服、未登入訪客）

**為什麼問這題**：不同角色看到的畫面常常不一樣，前端要知道有幾種版本要做。

### Q1.2 — 各角色的畫面差異
> 不同角色看到的畫面 / 能用的按鈕有沒有差異？例如：
> - 一般使用者只能看自己的清單
> - 管理員多一個「強制下架」按鈕
> - 未登入訪客看到一個「請先登入」的提示而不是內容

**為什麼問這題**：如果有差異，等於要做好幾個版本的同一個畫面，工時會差很多。

### Q1.3 — 未登入訪客
> 未登入的人能不能用這個功能？如果不能，他點進來會看到什麼？（轉去登入頁？顯示一個 banner？）

**為什麼問這題**：這是常被遺漏的情境，但實際很多使用者會從外部連結進來。

## 維度 2：User Flow Completeness（使用者流程完整度）

### Q2.1 — 從哪裡進入
> 使用者從哪裡進入這個功能？請列出所有入口：
> - 首頁的某個按鈕？
> - Email 通知的連結？
> - 推播通知點擊？
> - 直接打網址？
> - 其他功能跳過來？

**為什麼問這題**：每個入口可能要帶不同的初始資訊，不講清楚就會漏做。

### Q2.2 — 完整步驟
> 使用者從進入到完成這個功能，請描述完整的步驟（包含每一步看到什麼、按什麼）。

**為什麼問這題**：前端要把畫面串起來，沒有完整流程串不起來。

### Q2.3 — 中途取消 / 失敗
> 如果使用者：
> - 操作到一半關掉視窗
> - 中途點「取消」
> - 操作失敗

> 預期看到什麼？回到哪個畫面？已輸入的內容還在嗎？

**為什麼問這題**：流程的「異常出口」常被遺漏，但實際使用者經常會取消或失敗。

## 維度 3：Screen / View Inventory（畫面清單盤點）

### Q3.1 — 一共幾個畫面
> 這個功能會用到幾個畫面？請列出所有畫面，每個畫面寫一句它的目的。

**為什麼問這題**：前端要估時，畫面數量是最直接的指標。

### Q3.2 — 新畫面 vs 改既有畫面
> 每個畫面是「全新」還是「修改既有的某個畫面」？修改的話是改哪個既有畫面？

**為什麼問這題**：改既有畫面要避免影響別的功能，工時與風險不同。

### Q3.3 — 同畫面多種模式
> 有沒有「同一個畫面但長得不太一樣」的情況？例如：
> - 列表畫面 vs 列表搜尋後的結果畫面
> - 商品詳情頁的「上架中」vs「已下架」
> - 訂單列表的「全部」「待付款」「已完成」tab

**為什麼問這題**：這些常被當成一張畫面寫，但實際上需要不同設計。

## 維度 4：Interaction Granularity（互動細節顆粒度）

### Q4.1 — 互動元素清單
> 每個畫面上有哪些可以「點 / 填 / 選」的元素？請列出（按鈕、輸入框、下拉選單、開關、tab、分頁、排序、篩選）。

**為什麼問這題**：每個元素都是工，少列一個就少做一個，使用者體驗就缺一塊。

### Q4.2 — 點下去做什麼
> 每個元素點下去 / 操作後，預期發生什麼？
> - 切換到別的畫面？
> - 在原畫面跳出視窗（modal）？
> - 更新某個區塊（例如商品列表重新載入）？
> - 顯示成功 / 失敗訊息？

**為什麼問這題**：同一個按鈕的不同行為（跳新頁 vs 跳 modal）對前端是完全不同的工。

### Q4.3 — 什麼時候不能用
> 有沒有哪個元素「在某些情況下要變灰 / 不能按」？例如：
> - 表單沒填完的時候「送出」按鈕變灰
> - 庫存為 0 時「加入購物車」變灰
> - 還在處理中時所有按鈕暫時不能按

**為什麼問這題**：禁用狀態是常見漏寫的，但是體驗的關鍵。

## 維度 5：Visual Acceptance Criteria（視覺驗收條件）

### Q5.1 — 五種狀態的畫面
> 對於每個會載入資料的畫面，以下五種狀態長怎樣？
> 1. **載入中** — 等資料的時候顯示什麼？（轉圈圈？灰底框？）
> 2. **空的** — 完全沒資料時顯示什麼？（例如「您還沒有收藏任何商品」+ 一張示意圖 + 「去逛逛」按鈕）
> 3. **載入失敗** — 顯示什麼？（錯誤訊息？「重新載入」按鈕？）
> 4. **成功完成操作** — 怎麼回饋？（toast？跳轉？訊息列？）
> 5. **操作失敗** — 怎麼回饋？訊息文案是什麼？

**為什麼問這題**：這五種狀態都會發生，沒給就由前端自己編，編出來常跟 PM 預期不同。

### Q5.2 — 訊息文案
> 上面提到的成功 / 失敗訊息，實際文字是什麼？請給確切的文案，不要只說「顯示錯誤」。

**為什麼問這題**：文案是 PM 該寫的，不該由 RD 編。錯字 / 語氣不對的責任歸屬要清楚。

### Q5.3 — 手機 / 平板 / 桌機
> 這個功能在手機、平板、桌機上是否都要支援？三種尺寸的版面有什麼差異？

**為什麼問這題**：RWD 的工時可以是 1 倍到 3 倍不等，這題決定報價。

## 維度 6：Navigation & State Persistence（導航與狀態保留）

### Q6.1 — 畫面之間怎麼跳
> 各畫面之間怎麼切換？請描述。例如：
> - 列表頁 → 點商品 → 進詳情頁
> - 詳情頁 → 點「加入收藏」→ 留在原頁顯示 toast

**為什麼問這題**：跳新頁 vs 在原頁更新，是兩種完全不同的設計。

### Q6.2 — 按返回會怎樣
> 使用者按瀏覽器「返回」按鈕，預期看到什麼？例如：
> - 從詳情頁返回列表，篩選條件還在嗎？
> - 從付款頁返回購物車，已選的商品還在嗎？

**為什麼問這題**：返回行為是 UX 的關鍵，但常被忽略，做出來的話會被 QA / 使用者抱怨。

### Q6.3 — 重新整理頁面
> 使用者按 F5 / 重新整理時，預期：
> - 已選的篩選條件還在嗎？
> - 已填到一半的表單還在嗎？
> - 目前在第幾頁、捲到哪裡，還在嗎？

**為什麼問這題**：要不要保留狀態，影響很多技術選擇。

### Q6.4 — 網址能不能分享
> 使用者複製當前網址貼給朋友，朋友打開預期看到什麼？例如：
> - 列表頁的篩選條件 + 排序 + 第幾頁，是否要寫進網址讓朋友看到一樣的內容？
> - 還是不在乎，朋友打開就是預設狀態？

**為什麼問這題**：這個決定影響網址設計，越晚改成本越高。

---

## 模板使用方式

寫 PM feedback doc 時：
1. 先看每個維度給 0/1/2 分的（給 3 分的不需要問）
2. 從對應維度挑 1-3 個最關鍵的問題模板
3. 把模板中的通用名詞替換成 PM 文件的實際情境
4. 排到 feedback doc 的「逐題澄清清單」區塊
