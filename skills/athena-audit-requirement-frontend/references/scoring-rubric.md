# 前端視角 Scoring Rubric

PM 需求文件「前端可譯性」評分維度。對應 specformula Phase 01 / 06 / 07
（Activity Diagrams / Frontend Build / Integration）的反向推導 — 前端 RD 看完
PM 文件能否機械推出使用情境、畫面清單、互動元素、視覺驗收條件、導航行為。

## 評分尺度

每維度 0-3 分：

| 分數 | 含義 |
|------|------|
| **0** | 完全沒提，前端 RD 必須回頭問 PM 才能繼續 |
| **1** | 略有提及但不夠用，仍需大量補問 |
| **2** | 大致可推但有局部缺口，補 1-2 個問題即可 |
| **3** | 充分，前端 RD 可直接機械萃取，無需追問 |

總分 0-18。**對話中以自然語言陳述「N 個維度給得足、M 個維度需補」，不輸出 yaml score 欄位。**

> 不設「合格分數線」 — 本工具不是 gate，分數只是讓 PM 看見哪幾條缺。

---

## 維度 1：Actor / Persona Identification（角色 / 使用者識別）

**對應 specformula Phase**：01 (Requirement Analysis — Actor 合法性)

**追問點**：
- 文件是否點名「誰會用這個功能」？（一般使用者、付費會員、管理員、客服、訪客）
- 不同角色看到的畫面 / 能做的操作是否不同？
- 是否說明「未登入訪客」會看到什麼？

**評分判準**：
- 0：完全沒提使用者角色
- 1：泛泛提「使用者」但沒區分角色
- 2：點名主要角色但角色間差異不明
- 3：角色 + 各角色可看 / 可做的差異都齊

> 前端的差異：後端視角看「資料權限」，前端視角看「畫面差異與按鈕可見性」。同一個維度名但檢視角度不同。

---

## 維度 2：User Flow Completeness（使用者流程完整度）

**對應 specformula Phase**：01 (Activity Diagrams — entry → goal 流程)

**追問點**：
- 從使用者「進入這個功能」到「達成目標」的完整步驟是否描述？
- 流程中的分支（成功 / 失敗 / 中途取消）是否說明？
- 是否說明使用者「從哪裡來」（從首頁？從通知？從 email 連結？）

**評分判準**：
- 0：只描述某個畫面要長怎樣，沒提流程
- 1：流程主幹有，但分支與起點不明
- 2：主幹 + 分支有，但 entry 點不全
- 3：entry → 主幹 → 分支 → goal 都清楚

---

## 維度 3：Screen / View Inventory（畫面清單盤點）

**對應 specformula Phase**：06 (Frontend Build — pages 清單)

**追問點**：
- 這個功能會用到幾個畫面？每個畫面的目的是什麼？
- 同一個畫面有沒有多種模式（例如「列表畫面」與「列表的搜尋結果畫面」是同一張還是兩張）？
- 是否說明哪些是新畫面、哪些是改既有畫面？

**評分判準**：
- 0：完全沒提「會有什麼畫面」
- 1：點名一兩個畫面但沒列全
- 2：畫面清單齊但目的不明
- 3：畫面清單 + 每個畫面目的 + 新增 vs 修改的標註都齊

---

## 維度 4：Interaction Granularity（互動細節顆粒度）

**對應 specformula Phase**：06 (Frontend Build — UI components & events)

**追問點**：
- 每個畫面有哪些可互動的元素？（按鈕、輸入框、下拉選單、開關、tab、分頁）
- 每個互動元素點下去 / 操作後預期發生什麼？（彈出 modal？切換畫面？更新區塊？顯示提示？）
- 是否說明禁用狀態（什麼時候按鈕不能按 / 變灰）？

**評分判準**：
- 0：只描述畫面長怎樣，沒提互動
- 1：列了主要按鈕但沒說操作後行為
- 2：互動元素 + 行為有，但禁用狀態不明
- 3：互動元素、行為、禁用條件都齊

---

## 維度 5：Visual Acceptance Criteria（視覺驗收條件）

**對應 specformula Phase**：07 (Integration / Chrome Test Guard — UI 狀態驗收)

**追問點**：
- 是否說明各種「狀態」下畫面長怎樣：
  - **Loading** — 資料載入中（轉圈圈？skeleton？）
  - **Empty** — 沒有資料（例如收藏清單空空）
  - **Error** — 資料載入失敗
  - **Success** — 操作成功後的回饋（toast？跳轉？inline 訊息？）
  - **Failure** — 操作失敗的訊息
- 是否說明訊息文案（不只說「顯示錯誤」而是給實際文字）？
- 是否說明 RWD 行為（手機 / 平板 / 桌機）？

**評分判準**：
- 0：只給 happy 狀態畫面，其他狀態完全沒提
- 1：提到 loading / empty 其中一兩個
- 2：5 種狀態大致都提到但文案不齊
- 3：5 種狀態 + 文案 + RWD 都齊

---

## 維度 6：Navigation & State Persistence（導航與狀態保留）

**對應 specformula Phase**：06 (Frontend Build — routing) + 07 (Integration — 重新整理行為)

**追問點**：
- 畫面之間怎麼跳轉？（點按鈕？網址直接打？麵包屑？）
- 使用者按瀏覽器「返回」時預期看到什麼？
- 重新整理頁面後，使用者已填的內容 / 已選的篩選條件保不保留？
- 是否需要把當前狀態放到網址（例如 `?tab=收藏&page=2` 讓使用者可分享連結）？

**評分判準**：
- 0：完全沒提導航行為
- 1：主要跳轉提到了，但返回 / 重新整理沒提
- 2：跳轉 + 返回有，但狀態保留 / URL 共享不明
- 3：跳轉、返回、重新整理、URL 共享都齊

## 整體陳述範例（對話用）

```
6 個維度中：
- ✅ 給得足：Actor / Persona、User Flow、Screen Inventory
- 🟡 需補充：Interaction Granularity（按鈕禁用條件不明）、Visual Acceptance（empty / error 狀態未說明）、Navigation（重新整理後的狀態保留未說明）

整體看起來流程與畫面骨架清楚，但畫面細節（狀態、互動、導航）需要 PM 再補一輪。
```

> 範例只是參考，實際輸出依需求文件而定。**不要寫成 yaml**，純自然語言。

## 與後端視角的明確區隔

本 rubric 全部聚焦「畫面與流程」，**不評估資料 / API / 業務規則 / 驗證**。
若 PM 文件對後端原料也有缺，請另跑 `athena-audit-requirement-backend` 取得後端視角報告。

| 看的東西 | 前端維度 | 後端維度 |
|---------|---------|---------|
| 「使用者」 | 角色 / 畫面差異 | 帳號資料屬性 / 權限 |
| 「流程」 | 畫面跳轉 / 互動順序 | command/query 邊界 / 觸發者 |
| 「狀態」 | UI 狀態（loading/empty/error） | 資料狀態（草稿/已送出/已取消） |
| 「驗證」 | 輸入時的 UI 提示 | 後端格式 / 範圍 / 唯一性 |
| 「失敗」 | 錯誤訊息畫面 / 文案 | 例外處理 / 重試 / 回滾 |
