# PM Feedback 文件骨架（後端視角）

寫入路徑：`requirement-feedback/<slug>-backend.md`

> 這份文件給 PM 看。**全文使用業務語言，不出現 RD 術語**。
> 禁用詞：erm.dbml / api.yml / endpoint / payload / migration / schema 等等。

---

```markdown
# 需求 Audit Feedback — Backend 視角

- 需求名稱：<從 PM 文件標題或 slug 推>
- Audit 日期：<YYYY-MM-DD>
- 來源：<PM 文件路徑或「貼文字」>

## 一句話摘要

<把 PM 給的需求重述為一句話，讓 PM 確認 RD 的理解沒走偏>

## 整體觀察

<用 2-3 句話陳述：哪幾個維度給得清楚、哪幾個維度需要補。**不寫機器可解析欄位**，純自然語言。>

範例：
> 這份需求的「使用者要做什麼」寫得清楚，但「資料怎麼存」與「出錯怎麼辦」這兩塊還需要補一些細節。整體看 6 個重點裡有 4 個給得足，2 個需要 PM 再補一輪。

## ✅ 已給足的部分

逐項列點。每點一句陳述。

例如：
- 主要資料對象（使用者、訂單）已點名清楚
- 使用者主要動作（建立訂單、查詢訂單）邊界清楚
- 沒有外部系統依賴，整合點 N/A

## 🟡 需要補完的部分（精簡版）

每個維度一段，列出該維度為什麼判定為「需補」，**不重複下方逐題清單**。

例如：
- **資料驗證規則**：使用者要填的欄位中，哪些是必填、有什麼格式限制（例如電話幾位數、email 是否要驗證真實性）目前沒提到。
- **出錯時的處理**：「使用者餘額不足時要顯示什麼訊息」「同一筆訂單被點兩次提交怎麼辦」這類情境未說明。

## 💡 進階建議

未來可以這樣準備需求單，會讓後續開發更順：

例如：
- 寫需求單時可以準備一張「使用者填寫的欄位 → 必填/選填/格式限制」表格
- 對每個按鈕動作補一句「失敗時顯示什麼訊息」

## 補完後的下一步

1. PM 在每個「PM 補答」區塊填上回答
2. 把補完的版本貼回對話，或更新原 PM 需求文件
3. 重跑 audit：`/athena-dev-plugin:athena-audit-requirement-backend <path> --slug=<slug>`
4. 若整體變綠（多數維度 ✅），即可進入後端 RD 工程流程（`/athena-dev-plugin:athena-point` 評分後決定走 spec 或 build）

---

> 這份回饋是輔導建議，不阻擋任何工程流程。即使有題目沒補完，後端 RD 仍可以憑現有資訊開工，只是會在 build 過程中多回頭問 PM 幾次。
```
