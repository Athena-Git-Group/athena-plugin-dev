# PM-Friendly 提問模板（後端視角）

這份模板是給 PM 看的澄清問題範本。**全部用業務語言**，不出現 RD 術語。
寫 PM feedback doc 時，依當下需求改寫具體名詞（把「使用者」「商品」等替換成 PM 文件實際的對象）。

## 寫作原則

| 原則 | 範例 |
|------|------|
| **用業務語言** | ❌「erm.dbml 推不出來」→ ✅「使用者點擊『加入收藏』時，系統需要記住哪些資訊？」 |
| **問「情境」而非「欄位」** | ❌「請列出 user 的欄位」→ ✅「使用者註冊時，會留下哪些資料供之後查詢使用？」 |
| **附上「為什麼問這題」** | 讓 PM 理解這題不是吹毛求疵，是真的會卡到 |
| **留空白讓 PM 直接補** | 在問題下方留「PM 補答：____」 |

## 禁用詞

`erm.dbml` / `api.yml` / `endpoint` / `payload` / `migration` / `schema` / `nullable` / `cascade` / `actor model` / `command/query` / `CRUD` / `foreign key` / `unique constraint` / `enum`

替代寫法：

| RD 術語 | PM-friendly 說法 |
|---------|------------------|
| schema / 欄位 | 「需要記住的資訊」「使用者填寫的資料」 |
| endpoint / API | 「使用者操作」「系統動作」 |
| migration | 「資料庫變動」「上線後資料怎麼處理」 |
| nullable | 「可以不填嗎」 |
| unique constraint | 「會不會有重複」 |
| cascade | 「刪掉之後相關資料怎麼處理」 |
| enum | 「有哪些可能的選項」 |

## 維度 1：Entity Identifiability（實體可辨識性）

### Q1.1 — 主要資料對象
> 在這個功能中，使用者主要在操作哪些「東西」？（例如：商品、訂單、收藏紀錄、文章草稿）每個東西大概要記住哪些資訊？

**為什麼問這題**：後端要決定「這次要動到資料庫的哪幾個地方」，沒這個資訊就無法估時、無法切任務。

### Q1.2 — 對象之間的關係
> 這些「東西」之間有什麼關係？（例如：一個使用者可以有很多訂單；一張訂單只屬於一個使用者）

**為什麼問這題**：關係決定了查詢與權限（例如：A 使用者能不能看到 B 使用者的訂單）。

### Q1.3 — 屬性的擁有者
> 這些資訊是屬於使用者帳號的（換帳號就看不到），還是全站共用的？

**為什麼問這題**：影響資料怎麼放、權限怎麼設。

## 維度 2：State Transition Clarity（狀態流轉清晰度）

### Q2.1 — 狀態盤點
> 這個功能的主要對象（例如訂單）會經過哪些狀態？（例如：草稿、待付款、已付款、已出貨、已完成、已取消）

**為什麼問這題**：如果有狀態，後端要設計「合法轉換」與「禁止操作」的邏輯。

### Q2.2 — 誰能改狀態
> 每個狀態變化是由誰觸發？（使用者點按鈕？系統自動？管理員後台？）改完狀態後系統要做什麼？（寄信？通知？扣庫存？）

**為什麼問這題**：決定權限與連帶動作。

### Q2.3 — 不可逆轉狀態
> 有沒有哪個狀態進去就不能再回頭？（例如「已取消」之後不能再變回「處理中」）

**為什麼問這題**：避免上線後 PM 才說「咦這個本來不該能改回來」。

## 維度 3：API Boundary Definition（API 邊界定義）

### Q3.1 — 動作清單
> 在這個功能中，使用者會做哪些動作？請逐一列出（例如：建立訂單、查詢訂單、取消訂單、修改訂單收件人）。

**為什麼問這題**：每個動作對應一個後端動作，列清楚才能估時、才能拆 ticket。

### Q3.2 — 一個動作還是好幾件事
> 例如「儲存」這個按鈕，背後其實是一件事（純儲存）還是好幾件事一起做（儲存 + 寄通知 + 紀錄歷史）？

**為什麼問這題**：使用者看到的「一個按鈕」常常背後是好幾個動作，需要拆清楚才知道哪些可能會失敗、哪些可以重試。

### Q3.3 — 動作的觸發者
> 這個動作是只有使用者會主動觸發，還是系統定時也會觸發（例如每天凌晨自動結算）？或外部系統會通知（例如金流 webhook）？

**為什麼問這題**：影響後端要做的「入口」設計。

## 維度 4：Business Rule Completeness（業務規則完整度）

### Q4.1 — 一切順利時
> 一切順利的情況下，使用者操作這個功能的完整流程是什麼？預期看到什麼回饋？

**為什麼問這題**：建立 happy path baseline。

### Q4.2 — 出錯的情況
> 哪些情況算「出錯」？（例如：餘額不足、超過限額、權限不夠、商品已下架、輸入格式錯誤）每種出錯情況要顯示什麼訊息給使用者？

**為什麼問這題**：unhappy path 不寫，後端會自己編，上線後跟 PM 預期不符。

### Q4.3 — 邊界情況
> 以下情況要怎麼處理？
> - 兩個使用者同時搶同一個商品 / 名額
> - 操作到一半網路斷線
> - 操作到一半超過時間（例如付款 30 分鐘沒完成）
> - 資料是空的（例如收藏清單是空的時候畫面長怎樣）

**為什麼問這題**：edge case 是真的會發生，沒講就會變 bug。

## 維度 5：Data Validation Rules（資料驗證規則）

### Q5.1 — 必填 vs 選填
> 使用者要填寫的每個欄位，哪些是必填、哪些是選填？

**為什麼問這題**：必填規則影響資料庫設計與表單驗證。

### Q5.2 — 格式與範圍
> 每個欄位有什麼格式或範圍限制？例如：
> - 文字：最多幾個字
> - 數字：最小最大是多少
> - 日期：可以選未來日期嗎、最久幾個月前
> - 特殊格式：email、電話、身分證

**為什麼問這題**：這些規則決定使用者輸入時要不要擋，沒擋的話資料會髒。

### Q5.3 — 不能重複的資訊
> 有沒有哪些資訊是「不能重複」的？（例如同一個 email 不能註冊兩次、同一個商品在購物車裡只能有一筆）

**為什麼問這題**：唯一性約束影響資料庫與 UI 上的提示訊息。

### Q5.4 — 格式錯誤時的訊息
> 使用者輸入錯誤時，要顯示什麼訊息？

**為什麼問這題**：訊息文案是 PM 該寫的，不是 RD 該編的。

---

## 維度 6：Integration Points（整合點 / 外部依賴）

### Q6.1 — 依賴的外部系統
> 這個功能會用到哪些外部系統？（例如：金流、簡訊、email 寄件、Google 登入、第三方推播）每個用來做什麼？

**為什麼問這題**：外部系統等於「我們不能控制的失敗點」，要事先講好怎麼處理。

### Q6.2 — 連帶動作（Side Effects）
> 使用者完成這個操作後，系統還會自動做哪些事？（例如：寄通知信、寫 audit log、發推播、扣點數、觸發排程）

**為什麼問這題**：side effect 不講清楚，使用者重試時會發生重複扣款 / 重複寄信。

### Q6.3 — 外部系統失敗的處理
> 如果外部系統失敗（例如金流回應超時、簡訊送失敗），預期怎麼處理？
> - 自動重試幾次？
> - 顯示什麼訊息給使用者？
> - 是否要人工補單？

**為什麼問這題**：失敗處理是 PM 的業務決策，不是 RD 該自己決定的。

---

## 模板使用方式

寫 PM feedback doc 時：
1. 先看每個維度給 0/1/2 分的（給 3 分的不需要問）
2. 從對應維度挑 1-3 個最關鍵的問題模板
3. 把模板中的通用名詞替換成 PM 文件的實際情境
4. 排到 feedback doc 的「逐題澄清清單」區塊
